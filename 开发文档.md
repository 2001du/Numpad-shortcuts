## 1. 环境准备

### 1.1 安装 Node.js

**为什么需要 Node.js？**
- VSCode 插件基于 Node.js 运行环境
- 插件需要使用 npm/yarn 管理依赖
- 编译打包需要 Node.js 支持

```bash
# 检查 Node.js 版本（建议 16.x 或更高）
node --version

# 检查 npm 版本
npm --version
```

**为什么选择特定版本？**
- VSCode 插件引擎要求 Node.js 版本与 VSCode 版本兼容
- 较新的 Node.js 版本提供更好的性能和安全性

---

### 1.2 安装 TypeScript

**为什么使用 TypeScript？**
- VSCode 插件 API 提供了完整的 TypeScript 类型定义
- 编译时类型检查减少运行时错误
- 提供更好的代码提示和智能补全
- 团队协作时代码可读性更强

```bash
# 全局安装 TypeScript（可选，已在项目本地安装）
npm install -g typescript

# 验证安装
tsc --version
```

**TypeScript 的优势详解：**
```typescript
// 不使用 TypeScript - 运行时才能发现错误
const command = vscode.workspace.getConfiguration('numpad-shortcuts');
const num = command.get('num1'); // 如果配置不存在，返回值不明确

// 使用 TypeScript - 编译时就能发现类型错误
const command = vscode.workspace.getConfiguration('numpad-shortcuts');
const num: string = command.get('num1') || ''; // 类型明确，IDE 提供完整提示
```

---

### 1.3 安装 VSCode

**为什么需要 VSCode？**
- 官方提供插件开发工具和调试器
- 内置扩展开发助手 Extension API
- 支持 TypeScript 编译和调试

**插件开发必备扩展：**
- Extension Test Runner（测试运行器）
- ESLint（代码规范检查）

### 1.4 安装项目依赖

**为什么需要安装依赖？**
- `@types/vscode`: 提供 VSCode API 的 TypeScript 类型定义
- `typescript`: TypeScript 编译器
- 这些是开发依赖，不会打包到最终插件中

```bash

cd /workspace

# 安装依赖
npm install
# 或使用 yarn
yarn install
```

**package.json 中相关配置：**
```json
{
  "devDependencies": {
    "@types/vscode": "^1.80.0",  // VSCode API 类型定义

  }

```

**为什么这些是 devDependencies？**
- 它们只在开发阶段使用
- VSCode 运行时已经内置了所需模块
- 减小最终插件的体积

## 2. 项目结构详解

```
/workspace/
├── src/                        # 源代码目录
│   └── extension.ts            # 插件主入口文件
├── out/                        # 编译输出目录
│   ├── extension.js            # 编译后的 JavaScript 文件
│   └── extension.js.map        # Source Map（用于调试）
├── node_modules/               # 依赖包目录
├── package.json                # 项目配置文件（核心）
├── tsconfig.json               # TypeScript 编译配置
├── .vscodeignore               # 打包时忽略的文件
├── package-lock.json           # npm 锁定文件
├── yarn.lock                   # yarn 锁定文件
└── LICENSE                     # 开源许可证
```

---

### 2.1 目录结构设计原则

**为什么这样组织？**
1. **分离源码和编译输出**
   - `src/` 目录存放 TypeScript 源码
   - `out/` 目录存放编译后的 JavaScript
   - 这样可以清晰区分开发和生产代码

2. **使用 TypeScript 而非 JavaScript**
   - 类型安全
   - 更好的 IDE 支持
   - 编译时错误检查

3. **版本锁定文件**
   - `package-lock.json` 或 `yarn.lock` 确保依赖版本一致
   - 避免不同机器上安装不同版本的依赖

## 3. 核心配置文件解析

### 3.1 package.json - 插件配置核心

**为什么这是最重要的文件？**
- VSCode 通过这个文件识别插件
- 定义插件的所有功能和行为
- 控制插件的安装和加载时机

---

#### 3.1.1 基本信息

```json
{
  "name": "numpad-shortcuts",
  "displayName": "Numpad Shortcuts",
  "description": "Use numpad keys to run custom commands",
  "version": "0.0.1",
  "engines": {
    "vscode": "^1.80.0"
  },
  "categories": ["Other"]
}
```

**各字段说明：**
- `name`: 插件唯一标识符（发布到市场后不能修改）
- `displayName`: 在扩展市场中显示的友好名称
- `description`: 插件功能的简短描述
- `version`: 语义化版本号，格式为 `主版本.次版本.修订号`
- `engines.vscode`: 兼容的 VSCode 最低版本
  - 为什么需要这个字段？
  - 某些 API 只在特定版本后可用
  - 防止用户在不兼容的 VSCode 中安装插件
- `categories`: 插件分类（如 "Other", "Programming Languages" 等）
  - 为什么需要分类？
  - 帮助用户在市场中更快找到插件

---

#### 3.1.2 激活事件

```json
{
  "activationEvents": ["*"]
}
```

**为什么需要激活事件？**
- VSCode 采用按需加载机制
- 插件不会在启动时全部加载，避免影响性能
- `*` 表示在任何情况下都激活
  - 为什么使用 `*`？
  - 这个插件需要响应全局快捷键
  - 不论用户在做什么，按下数字键都应该响应
  - 节省激活配置的复杂度

**其他激活事件示例：**
```json
{
  "activationEvents": [
    "onLanguage:javascript",  // 打开 JS 文件时激活
    "onCommand:myextension.hello",  // 执行命令时激活
    "onStartupFinished",     // VSCode 启动完成后激活
    "onView:myView",         // 打开特定视图时激活
    "*"                      // 始终激活
  ]
}
```

**为什么不同项目使用不同激活事件？**
- 优化性能：只在需要时加载插件
- 减少内存占用：未使用的插件不占用资源
- 提升启动速度：避免所有插件都同时加载

---

#### 3.1.3 主入口文件

```json
{
  "main": "./out/extension.js"
}
```

**为什么指向编译后的文件？**
- VSCode 运行环境只支持 JavaScript
- TypeScript 需要编译后才能运行
- `main` 字段告诉 VSCode 入口文件的位置

---

#### 3.1.4 贡献点（Contribution Points）

**什么是贡献点？**
- 插件向 VSCode 扩展功能的接口
- 通过 `contributes` 字段定义
- 这是插件与 VSCode 集成的核心机制

**为什么需要贡献点？**
- 实现插件功能的声明式配置
- VSCode 自动注册和管理
- 统一的扩展机制，避免直接操作 VSCode 内部

---

#### 3.1.5 命令注册

```json
{
  "contributes": {
    "commands": [
      { "command": "numpad-shortcuts.num1", "title": "Run Numpad 1 Task" },
      { "command": "numpad-shortcuts.num2", "title": "Run Numpad 2 Task" },
      ...
    ]
  }
}
```

**字段说明：**
- `command`: 命令的唯一标识符（在代码中调用）
  - 格式：`插件名.命令名`
  - 为什么需要唯一标识？
  - 避免不同插件的命令冲突
  - 在代码中通过这个 ID 调用命令
- `title`: 命令在命令面板（Ctrl+Shift+P）中显示的名称
  - 为什么需要友好的标题？
  - 用户搜索时更容易找到
  - 提升用户体验

**为什么要注册命令？**
- 让命令可以在命令面板中使用
- 可以为命令绑定快捷键
- 可以为命令添加到菜单或工具栏

---

#### 3.1.6 键盘快捷键

```json
{
  "contributes": {
    "keybindings": [
      { 
        "command": "numpad-shortcuts.num1", 
        "key": "numpad1", 
        "when": "editorTextFocus || terminalFocus" 
      },
      ...
    ]
  }
}
```

**字段说明：**
- `command`: 要执行的命令 ID
- `key`: 按键标识符
  - `numpad1`: 数字键盘的 1 键
  - 支持的组合：`Ctrl+S`, `Alt+Shift+F` 等
- `when`: 激活条件（Context）
  - `editorTextFocus`: 编辑器获得焦点
  - `terminalFocus`: 终端获得焦点
  - 为什么需要条件？
  - 避免在不需要的地方触发快捷键
  - 防止快捷键冲突

**为什么需要快捷键？**
- 提升操作效率
- 减少鼠标使用
- 符合用户习惯

---

#### 3.1.7 配置选项

```json
{
  "contributes": {
    "configuration": {
      "type": "object",
      "title": "Numpad Shortcuts",
```

- `default`: 默认值

  - 首次使用时有合理的默认行为
      - `description`: 配置项的描述
  - 在设置界面显示，帮助用户理解

**配置项的访问方式：**
```typescript
const config = vscode.workspace.getConfiguration('numpad-shortcuts');
```

## 4. 代码实现详解

### 4.1 extension.ts - 插件主入口

**为什么这是入口文件？**
- VSCode 加载插件时执行 `activate` 函数
- 插件卸载时执行 `deactivate` 函数
- 这两个函数是插件生命周期的一部分

---

### 4.2 activate 函数 - 激活逻辑

```typescript
export function activate(context: vscode.ExtensionContext) {
    console.log('Numpad Shortcuts extension is now active');

    // Register commands for numpad 0-9
    for (let i = 0; i <= 9; i++) {
        const disposable = vscode.commands.registerCommand(
            `numpad-shortcuts.num${i}`, 
            () => {
                executeNumpadCommand(i);
            }
        );
        context.subscriptions.push(disposable);
    }
}
```

**参数说明：**
- `context: vscode.ExtensionContext`
  - VSCode 传给插件的上下文对象
  - 包含插件运行时的信息和工具
  - 为什么需要这个对象？
    - 管理插件的生命周期
    - 提供存储全局变量的空间
    - 管理 disposables（清理资源）

---

**核心逻辑：**

1. **循环注册 0-9 的命令**
   ```typescript
   for (let i = 0; i <= 9; i++) {
       const disposable = vscode.commands.registerCommand(
           `numpad-shortcuts.num${i}`, 
           () => executeNumpadCommand(i)
       );
   }
   ```
   - 为什么用循环？
     - 避免重复代码
     - 易于维护和扩展
   - 命令 ID 必须与 package.json 中一致

2. **registerCommand 返回 Disposable**
   ```typescript
   const disposable = vscode.commands.registerCommand(...);
   context.subscriptions.push(disposable);
   ```
   - **为什么需要 Disposable？**
     - VSCode 使用资源管理模式
     - 插件卸载时需要清理注册的资源
     - 避免内存泄漏
   - **为什么 push 到 subscriptions？**
     - 让 VSCode 自动管理这些资源
     - 插件卸载时自动调用 dispose()
     - 无需手动清理

3. **箭头函数的作用**
   ```typescript
   () => { executeNumpadCommand(i); }
   ```
   - 闭包捕获变量 `i`
   - 为什么用箭头函数？
     - 保持 `this` 指向
     - 语法简洁

---

### 4.3 executeNumpadCommand 函数 - 执行命令

```typescript
function executeNumpadCommand(num: number) {
    // 1. 读取配置
    const config = vscode.workspace.getConfiguration('numpad-shortcuts');
    const command = config.get<string>(`num${num}`);
    
    // 2. 验证配置
    if (!command || command.trim() === '') {
        vscode.window.showWarningMessage(
            `Numpad ${num} is not configured. ` +
            `Please set it in Settings → Extensions → Numpad Shortcuts`
        );
        return;
    }

    // 3. 获取或创建终端
    const terminal = vscode.window.activeTerminal || 
                     vscode.window.createTerminal('Numpad Shortcut');
    terminal.show();
    
    // 4. 执行命令
    terminal.sendText(command);
    
    // 5. 显示提示
    vscode.window.showInformationMessage(`Executed: ${command}`);
}
```

---

#### 4.3.1 读取配置

```typescript
const config = vscode.workspace.getConfiguration('numpad-shortcuts');
const command = config.get<string>(`num${num}`);
```

**为什么使用 getConfiguration？**
- 获取用户在 settings.json 中配置的值
- 自动处理配置的继承和优先级
- 支持工作区和全局配置

**为什么指定类型 `<string>`？**
- TypeScript 泛型，提供类型提示
- 编译时类型检查
- IDE 智能补全

**配置读取规则：**
1. 工作区配置（`.vscode/settings.json`）优先
2. 用户配置（用户 settings.json）次之
3. 默认值（package.json）最后

---

#### 4.3.2 验证配置

```typescript
if (!command || command.trim() === '') {
    vscode.window.showWarningMessage(...);
    return;
}
```

**为什么需要验证？**
- 避免执行空命令
- 提供友好的错误提示
- 引导用户配置插件

**为什么使用 showWarningMessage？**
- VSCode 提供的标准通知方式
- 不阻塞用户操作
- 可以点击跳转到设置

---

#### 4.3.3 获取或创建终端

```typescript
const terminal = vscode.window.activeTerminal || 
                 vscode.window.createTerminal('Numpad Shortcut');
terminal.show();
```

**为什么用 || 运算符？**
- 如果已有活动的终端，使用它
- 否则创建新的终端
- 避免创建过多终端窗口

**为什么调用 show()？**
- 新创建的终端默认隐藏
- show() 显示终端面板
- 确保用户能看到命令执行

---

#### 4.3.4 执行命令

```typescript
terminal.sendText(command);
```

**为什么用 sendText？**
- 在终端中输入并执行命令
- 模拟用户手动输入
- 支持任何终端命令

---

#### 4.3.5 显示提示

```typescript
vscode.window.showInformationMessage(`Executed: ${command}`);
```

**为什么需要提示？**
- 确认命令已执行
- 提供视觉反馈
- 增强用户体验

---

### 4.4 deactivate 函数 - 清理资源

```typescript
export function deactivate() {}
```

**为什么函数是空的？**
- 这个插件不需要额外清理
- context.subscriptions 中的资源会自动清理
- 显式声明函数是良好的实践

**什么时候需要清理？**
- 关闭打开的文件
- 停止后台任务
- 保存临时数据
- 取消事件监听

## 5. 开发流程

### 5.1 编译代码

**为什么需要编译？**
- VSCode 不支持直接运行 TypeScript
- 需要编译为 JavaScript
- 生成 Source Map 便于调试

```bash
# 编译一次
npm run compile
# 或
tsc -p ./

# 监听模式（自动编译）
npm run watch
# 或
tsc -watch -p ./
```

**编译过程：**
1. 读取 `tsconfig.json` 配置
2. 编译 `src/` 目录下的 TypeScript
3. 输出到 `out/` 目录
4. 生成 `.js` 和 `.js.map` 文件

**为什么使用监听模式？**
- 开发时频繁修改代码
- 自动编译提高效率
- 减少手动操作

---

### 5.2 加载插件 - 开发模式

**为什么需要开发模式？**
- 不需要每次都打包安装
- 直接加载源码进行测试
- 支持热重载

**方法一：使用 VSCode 调试**
1. 打开项目
2. 按 `F5` 或点击"Run Extension"
3. VSCode 会打开新的 Extension Development Host 窗口
4. 在新窗口中测试插件

**为什么推荐这个方法？**
- 集成调试功能
- 可以设置断点
- 查看控制台输出
- 快速迭代

---

**方法二：命令行加载**
```bash
# 安装 vsce（VSCode Extension Manager）
npm install -g vsce

# 打包插件
vsce package

# 安装插件到 VSCode
code --install-extension numpad-shortcuts-0.0.1.vsix
```

**什么时候使用这个方法？**
- 需要在不同的 VSCode 实例中测试
- 分发给他人测试
- 准备发布

---

### 5.3 配置插件

## 8. 核心概念总结

### 8.1 插件生命周期

```
安装 → 激活(activate) → 运行 → 停用(deactivate) → 卸载
```

**为什么理解生命周期？**
- 在正确的时机做正确的事
- 避免资源泄漏
- 提高插件性能

---

### 8.2 资源管理

**Disposables 模式：**
```typescript
const disposable = vscode.commands.registerCommand(...);
context.subscriptions.push(disposable);
```

**为什么使用这个模式？**
- 自动清理资源
- 避免内存泄漏
- VSCode 自动管理生命周期

**其他 Disposable 示例：**
```typescript
// 事件监听
const listener = vscode.workspace.onDidChangeTextDocument(...);
context.subscriptions.push(listener);

// 状态栏项目
const statusItem = vscode.window.createStatusBarItem(...);
context.subscriptions.push(statusItem);
```

---

### 8.3 命令系统

**命令注册流程：**
1. 在 `package.json` 中声明命令
2. 在 `activate` 中注册命令处理器
3. 通过快捷键或命令面板触发

**为什么这样设计？**
- 声明式配置
- 统一的管理方式
- 支持多种触发方式

---

### 8.4 配置系统

**配置层级：**
```
默认值（package.json）→ 用户设置 → 工作区设置
```

**为什么需要层级？**
- 不同项目可能有不同配置
- 用户可以覆盖默认值
- 工作区配置不影响全局设置

## 10. 常见问题

### Q1: 为什么插件不生效？

**检查清单：**
- [ ] 是否按 F5 启动了调试模式？
- [ ] extension.js 是否存在（是否编译）？

- VSCode 没有终端权限

- 尝试手动打开终端
- 查看开发者工具日志
